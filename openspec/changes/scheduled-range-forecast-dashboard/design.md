## Context

GoldFXGraph 已经具备 XAUUSD 日线研究 workflow、forecast 持久化和 Vue 3 Dashboard，但现在的使用方式还不够“自动化研究台”：页面上还有手动刷新入口，后端也存在可被显式触发的研究运行接口，调度逻辑分散在多个地方，用户也无法直接看到当前是不是正在跑 agent、跑到了哪一步。

这次变更的核心目标，是把系统收束成一个统一的研究循环：每 15 分钟自动执行一次市场数据获取、当日最新数据更新和多智能体分析，并把当前执行状态、最近完成时间和最终 forecast 统一回传给前端。另一方面，forecast 的方向输出要从“单点方向”升级为“固定时间窗口的区间判断”，这样首页就能直接告诉用户短中期不同阶段偏多、偏空还是震荡。

## Goals / Non-Goals

**Goals:**

- 取消前端手动刷新和后端手工触发研究运行。
- 统一所有定时任务到一个 15 分钟自动调度流程。
- 在前端展示最新执行时间、运行中状态和各 agent 状态。
- 将 forecast 输出改为固定时间窗口的方向区间。
- 保留现有分析原因列表、K 线图、新闻和情绪模块。
- 首页首屏优先展示当前价格、市场方向和交易建议字段。

**Non-Goals:**

- 不引入自动交易、真实下单、券商集成或 broker 连接。
- 不做完整的多模型路由系统。
- 不重做现有 K 线图组件的视觉或交互逻辑。
- 不把市场新闻和情绪模块改造成新功能，只保持现状。

## Decisions

### 1) 采用单一统一调度器，而不是多个并行定时任务
统一的 15 分钟调度器会成为唯一的自动执行入口，按固定顺序执行全部研究步骤：获取最新黄金数据、更新当日数据快照、执行多智能体分析、持久化结果和运行状态。这样可以避免多个定时任务相互抢占、重复写入或前端看到不同步状态。

**为什么这样做：**
- 用户明确要求“所有定时任务统一在一起执行”。
- 统一流水线更容易展示“当前执行到哪一步”。
- 运行状态和最终结果共享同一个 run，便于追踪。

**备选方案：**
- 保留多个独立定时器：不采用，因为会造成状态分裂和调试复杂度上升。
- 前端轮询多个接口拼装状态：不采用，因为职责会过度分散。

### 2) 使用固定时间窗口表达方向，而不是自由文本预测天数
forecast 最终结果将携带一组固定窗口的方向区间，例如 `0-3天`、`3-5天`、`6-15天`、`15天后`。agent 在规划阶段会针对每个窗口给出方向、强度、理由和置信度，并可额外给出一个补充观察窗口。

**为什么这样做：**
- 固定窗口更适合前端稳定展示和历史对比。
- 用户已经确认不做“单独未来 3 天预测”，而是做“未来若干天的方向区间”。
- 结构化窗口比自由文本更利于后续排序、过滤和样式统一。

**备选方案：**
- 让 agent 每次自定义窗口长度：不采用，因为历史记录和 UI 会变得不稳定。
- 只输出一条总方向，再靠文本解释窗口：不采用，因为会弱化结构化展示。

### 3) 运行状态要落库，而不是只放内存
统一调度的运行状态会作为结构化数据保存，至少包含：
- `started_at`
- `completed_at`
- `current_stage`
- `agent_statuses`
- `last_error`

这样前端在刷新后仍然能看到最近一次执行发生了什么，而不会因为进程重启丢失状态。

**为什么这样做：**
- 用户要求前端展示最新执行时间和 agent 状态。
- 只靠内存无法跨刷新和跨重启保留信息。

**备选方案：**
- 仅用内存态：不采用，因为无法满足可追溯性。

### 4) 首屏重排为“价格 + 方向 + 区间 + 交易建议”
首页第一屏会优先展示当前价格、当前总方向、时间窗口方向区间、入场/止盈/止损、最新执行时间和运行状态。原有的多 agent 分析原因列表、K 线图、新闻与情绪部分继续保留，但不再占据首屏最关键位置。

**为什么这样做：**
- 用户最先想看到的是“现在怎么看、怎么做、系统跑到哪了”。
- 交易研究字段必须结构化可扫读，不能藏在长段落中。

### 5) 保持现有分析模块不变，只调整输出组织方式
技术、宏观、新闻、风险以及现有的原因列表继续保留，K 线图和市场新闻/情绪模块也保持不变。变化重点是：
- 增加统一调度状态
- 改变方向表达方式
- 重排首页信息层级

这能把变更控制在用户真正关心的范围内，避免无关重构。

## Data / API Shape

建议的新增或扩展结构包括：

- `ForecastWindowDirection`
  - `window_label`
  - `direction`
  - `strength`
  - `confidence`
  - `reason`
- `SchedulerRunStatus`
  - `status`
  - `started_at`
  - `completed_at`
  - `current_stage`
  - `agent_statuses`
  - `last_error`
- `ForecastResult`
  - 保留当前价格、OHLC、入场、止盈、止损、持有周期、风险摘要和免责声明
  - 追加窗口化方向区间字段

前端通过 typed service 读取最新 forecast 和最新调度状态，页面只负责展示，不负责触发运行。

## Risks / Trade-offs

- 统一调度器如果卡在某一步，会影响后续整个 15 分钟循环 → 需要清晰状态和失败恢复策略。
- 固定窗口可能不如自由窗口灵活，但更适合研究对比和前端稳定展示。
- 运行状态落库会增加一点持久化开销，但换来更好的可追溯性和可观测性。

## Migration Plan

1. 先补充运行状态和窗口化 forecast 的数据模型。
2. 再把统一调度器接到后端启动流程中，替换分散的手动触发入口。
3. 然后改造 workflow 输出，让 agent 按固定窗口生成方向区间。
4. 最后重排前端首屏，移除手动刷新，增加调度状态展示。
