## Context

GoldFXGraph 现在的实时报价已经统一到 TradingView，但 completed daily bars 的补齐链路仍然是独立的运行时维护逻辑。当前数据库里的最新完成日线可能会因为历史补数中断、外部源返回未完成日线或维护任务未按时执行而停住。用户要求进一步收紧边界：启动时必须先补齐到最新完成交易日，失败就阻止服务启动；运行中跨日后也必须继续追平缺口；研究前还要再做一次强校验，避免用过期日线做 forecast。

## Goals / Non-Goals

**Goals:**
- 启动时先补齐 completed daily bars，补齐失败直接失败退出。
- 运行时维护任务持续追平缺口，跨日后自动补齐遗漏交易日。
- 统一以 TradingView 历史日线作为补齐来源，不再允许假数据、未完成日线或推测值写入数据库。
- 研究前先检查日线是否追平；未追平时先补齐，失败则拒绝研究。

**Non-Goals:**
- 不改 realtime quote 的 TradingView-only 方案。
- 不引入新的交易执行、自动下单或券商集成。
- 不在这次变更里重新设计历史图表 UI。
- 不把 CSV 重新变成运行时补齐来源。

## Decisions

### 1. 启动阶段复用同一套维护逻辑，且失败即阻断启动
我们不会为启动单独写一套“只检查不补齐”的旁路逻辑，而是直接复用现有维护路径：先计算当前应有的最新完成交易日，再检查数据库最新 bar 是否追平，若未追平则执行补齐。任何补齐错误都会抛到启动流程，让应用不进入可用状态。

**备选方案**
- 只在启动时做检查，不补齐：实现更简单，但会让已知缺口持续存在，不符合用户要求。
- 启动后异步补齐：可以尽快提供服务，但会让前端和研究流程短时间内看到旧数据，和“绝不允许假数据”冲突。

### 2. completed daily bar 的补齐只认 TradingView 历史日线
补齐来源将统一到 TradingView 的历史日线抓取逻辑。补齐时只接受完整、合法、已完成的日线；如果 TradingView 返回的是当前未完成交易日、OHLC 不合法或缺字段，就直接跳过或失败，不写入数据库。

**备选方案**
- TradingView 主、Yahoo 备：补齐成功率可能更高，但会重新引入第二来源，和“统一口径”冲突。
- 继续沿用 Yahoo：当前已经证明 Yahoo 会返回未完成日线并触发校验失败，不能满足真实性要求。

### 3. 研究前加一次市场数据强校验，复用维护逻辑
研究 workflow 在 `tool_load_market_data` 之前先调用一遍同样的维护/补齐逻辑，确保数据库最新完成日线已经追平。如果未追平且补齐失败，workflow 必须返回受控错误，而不是继续生成看起来正常、实际基于旧数据的 forecast。

**备选方案**
- 只依赖启动时的补齐：如果服务运行中跨日后未及时更新，研究仍可能拿到旧数据。
- 让 workflow 自己猜测是否该补齐：会造成维护逻辑分叉，不利于测试和排障。

### 4. 数据持久化保持幂等，但不允许用“补洞式推测值”
补齐任务仍然采用数据库 upsert，保证重复执行不会写出重复行。但这只是幂等性，不代表允许写入任何推测值。只有 TradingView 返回、且通过校验的 completed daily bar 才能进入数据库。

## Risks / Trade-offs

- [TradingView 页面结构变化或临时不可用] → 启动会失败，研究也会失败；但这是接受的代价，因为比写入假数据更安全。
- [启动变慢] → 首次启动或长时间离线后重启可能需要补很多天的数据；可通过分批查询和幂等 upsert 降低影响。
- [研究可用性下降] → 在数据源短暂不可用时，系统会更保守地拒绝服务；这是为了保证研究结论真实可信。
- [历史补齐范围较大时日志噪声增加] → 通过明确的维护结果与错误信息区分“no-op / written / failed”。

## Migration Plan

1. 部署新的 TradingView 历史补齐逻辑，并将启动流程改为先补齐后启动。
2. 首次启动时如果数据库存在缺口，自动补齐到最新完成交易日。
3. 若 TradingView 暂时不可用，启动直接失败，由部署层重试或回滚到上一版本。
4. 研究 workflow 和 `research-run` 复用同一套补齐/强校验逻辑，避免启动后跨日产生新的旧数据研究。
5. 迁移完成后，清理 Yahoo 历史补齐路径和与“未完成日线可写库”有关的旧测试分支。

## Open Questions

- 启动时的补齐是否需要设定最大重试次数，还是完全交给部署层重试。
- TradingView 历史日线在极端情况下是否允许一次补一段再逐日验证，还是必须一次性完成整段缺口。
